## 4. The Deploy Scoring Model

Deploy uses a two-penalty model applied to `TileScore.rangeScore` during `OnCollect`.

### Penalty 1 — Range distance penalty

Applied per candidate tile in `tileDict`, once per candidate origin tile in the proximity candidate set:

```
rangeScore -= WeightsConfig[+0xcc] * distanceResult * tileScore.secondaryMovementScore
```

Where:
- `WeightsConfig[+0xcc]` — a float weight (name unresolved; referred to as `rangePenaltyScale` here)
- `distanceResult` — return value of `FUN_18053feb0(tileCoords, pathList)` — an int, likely path distance or tile count
- `tileScore.secondaryMovementScore` — `TileScore.+0x24`, the secondary movement score for this tile

This penalizes tiles that are costly to reach from candidate origin positions.

### Penalty 2 — Ally proximity penalty

Applied per ally team member that is "set up" (`Actor.+0x50 != 0`) and within 6 tiles of the candidate position:

```
for each ally where ally.isSetUp and ally.currentTile != null:
    distance = TileDistance(ally.currentTile, candidatePosition)
    if distance < 6.0:
        rangeScore -= (6.0 - distance) * WeightsConfig[+0xd0]
```

Where:
- `WeightsConfig[+0xd0]` — a float weight (name unresolved; referred to as `allyProximityPenaltyScale` here)
- The maximum penalty per ally is `6.0 * WeightsConfig[+0xd0]` (when distance = 0)
- The penalty decreases linearly to 0 at distance = 6

**Design intent:** Prevents multiple units from deploying to the same tile cluster. Deployed units are spread across the battlefield.

## 5. Full Deploy Lifecycle

```
OnCollect(self, strategy, tileDict)
│
├── GUARD: strategyMode != 0 → return false (behavior suppressed by strategy)
├── GUARD: Collect candidate deploy tiles of type 2 via ProximityData
├── GUARD: candidateList.Count == 0 → return false (nowhere to deploy)
│
├── For each candidateOriginTile:
│   └── FUN_1806343a0(entityInfo, tileCoords, pathList, ...) → fills pathList with TileScore entries
│
└── For each tileScore in tileDict:
    ├── Apply range distance penalty to tileScore.rangeScore
    └── For each set-up ally within 6 tiles: apply proximity penalty to tileScore.rangeScore

OnEvaluate(self, strategy) → int
│
├── GUARD: strategyMode != 0 → return 0
├── GUARD: m_IsDone → return 0
├── bestTile = GetHighestTileScore(self)
├── actorCurrentTile = strategy.GetActorTile()
├── IF bestTile.tile != actorCurrentTile:
│   ├── self.m_TargetTile = bestTile.tile
│   └── return 1000
└── ELSE (already at best tile):
    ├── agentContext.field_0x50 = 1   // signal deploy-complete to context
    ├── self.m_IsDone = true
    └── return 0

GetHighestTileScore(self) → TileScore*
│
└── Iterate tileDict, return TileScore with highest FUN_180740f20() score

OnExecute(self, actor) → bool
│
├── IF NOT self.field_0x1c (no target set — base Behavior flag):
│   └── return actor.field_0x15f == 0
└── ELSE:
    ├── IF m_TargetTile is occupied: set flags |= 1
    ├── FUN_1805e03b0(actor, m_TargetTile, &flags, 3, 0)  // issue move command
    ├── agentContext.field_0x50 = 1   // signal deploy-complete
    ├── self.m_IsDone = true
    └── return actor.field_0x15f == 0
```

## 9. Key Inferences and Design Notes

**Deploy is a positioning pre-requisite, not a combat behavior.** It scores a fixed 1000 when it has an unvisited target tile. This hardcoded priority means Deploy will always win over lower-scored combat behaviors when the unit is not yet in position. The system relies on `m_IsDone` to prevent it from firing after completion.

**The ally proximity penalty spreads deployed units.** The linear falloff over 6 tiles means the penalty is strongest when allies are adjacent and zero when they are 6+ tiles apart. This produces a soft spacing constraint between deployed units — they will naturally separate unless forced together by terrain.

**The "set up" check on allies is load-bearing.** The proximity penalty only applies to allies where `Actor.+0x50 != 0`. Allies that have not yet deployed or that are still mobile do not count. This prevents the penalty from incorrectly pushing units away from allies who will not be stationary.

**`agentContext + 0x50` write requires revisiting.** Previously this offset was labelled as `behaviorConfig*` (a pointer). However `OnEvaluate` and `OnExecute` both write byte value 1 to this location directly, which is inconsistent with it being a pointer. Either the label was wrong (it is a bool/byte field, not a pointer) or the write is intentionally stomping the low byte of a pointer with a flag. This is NQ-42 and should be resolved before collation finalises the AgentContext layout.
